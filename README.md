# 仿制 B 站项目介绍与技术分析

项目效果展示：【【基于GoLang开发的仿制B站的个人项目展示】】 https://www.bilibili.com/video/BV1TdL863E2h/?share_source=copy_web&vd_source=5248d9c426aa7a79c98b2dfe8e831f58

## 1. 项目概览

这个项目是一个面向视频内容场景的仿 B 站平台，采用前后端分离方式开发，整体目标不是只做静态页面，而是尽量把一个视频网站的核心链路串起来，包括用户认证、视频上传、转码、播放、评论、弹幕、点赞、收藏、通知和个人中心等模块。

从当前仓库的实际代码来看，它已经具备一个中小型视频平台的主干功能，尤其在“视频上传后异步转码再播放”的链路上，已经不只是页面模拟，而是把对象存储、消息队列、转码工具和数据库状态更新连接到了一起。前端在视觉上采用了统一的赛博科技风格，后端则采用相对清晰的 Controller / Service / DAO 分层结构，整体有比较明显的课程设计、项目展示和技术答辩价值。

这份文档不是 README 的替代品，而是一份基于当前源码观察结果整理出的“项目介绍 + 技术实现分析”融合稿。文中所有结论都以仓库中的真实代码和配置为依据，不把未完全落地的能力写成已完成成果，也不会直接复用现有根目录文档中的已有表述。

## 2. 项目定位与整体架构

### 2.1 项目定位

如果从项目用途来判断，这个项目更适合被理解为一个“具备工程化思路的视频平台练习项目”，而不是一个只有前端界面的仿站作品。它既包含内容网站常见的基础能力，也引入了后端服务开发中比较典型的基础设施：

- MySQL 负责核心业务数据持久化。
- Redis 负责点赞等高频状态缓存。
- MinIO 负责视频、封面、头像等文件对象存储。
- RabbitMQ 负责视频转码任务异步解耦。
- FFmpeg 负责视频转码与封面截帧。
- WebSocket 负责弹幕的实时广播。

这意味着项目的重点并不只是“看起来像 B 站”，而是试图复现一个视频平台最关键的业务闭环。

### 2.2 源码观察结论

根据当前仓库可以明确得出以下结论：

- 项目采用前后端分离架构。
- 前端使用 `Vue 3 + Vite + TypeScript`，并配合 `Pinia`、`Vue Router`、`Axios` 和 `DPlayer`。
- 后端使用 `Go + Gin + GORM`，数据库为 MySQL。
- 视频上传后会先写入 MinIO，再通过 RabbitMQ 投递转码任务，由消费者调用 FFmpeg 异步转码，最后回写数据库中的转码状态和播放地址。
- 视频点赞使用 Redis 维护用户点赞状态和点赞计数，并通过定时任务同步数据库中的统计字段。
- 弹幕系统通过 WebSocket 房间广播实现实时推送。
- 配置文件中包含数据库、Redis、RabbitMQ、MinIO 和 JWT 的明文敏感信息，这应该被视为改进项，而不是项目优势。
- 前端部分页面存在占位数据、硬编码统计值或未完全接入的视图文件，因此项目当前状态更接近“主链路已打通，部分细节仍需补完”。

### 2.3 整体架构说明

从运行形态来看，项目可以拆成四个层面：

| 层面 | 主要职责 | 当前实现 |
| --- | --- | --- |
| 前端展示层 | 页面渲染、路由切换、用户交互、请求接口 | `bilibili-frontend` |
| 后端业务层 | 用户、视频、评论、弹幕、通知等业务处理 | `bilibili-backend` |
| 基础设施层 | MySQL、Redis、MinIO、RabbitMQ、FFmpeg | 由后端配置与工具模块接入 |
| 异步处理层 | 转码消费、状态更新 | `consumer/transcode.go` |

在请求路径上，一个典型流程大致如下：

1. 前端通过 Axios 调用 `/api/v1/...` 接口。
2. Gin 接收请求并经过 CORS、中间件鉴权等处理。
3. Controller 负责参数接收和结果返回。
4. Service 负责业务编排，例如上传文件、投递转码任务、触发通知、写入历史。
5. DAO 负责通过 GORM 与 MySQL 交互。
6. 部分模块再分别访问 Redis、MinIO、RabbitMQ 或 WebSocket Hub。

这个结构对于学习型项目来说是比较清晰的，也便于后续继续扩展。

## 4. 各功能的技术实现方式

下面按统一模板，分别介绍各核心功能当前是如何落地的。

### 4.1 用户认证与个人中心

#### 功能作用

这一部分负责用户身份建立和基础账户管理，是整个平台所有需要登录功能的入口，包括登录、注册、登录态保持、个人资料编辑、密码修改和头像上传。

#### 前端表现

前端使用 `LoginView.vue` 承担登录与注册双模式页面，用户可以在同一页面切换“登录 / 注册”标签。登录成功后，令牌会被保存在浏览器本地存储中，刷新页面后由 `Pinia` 中的 `user` store 恢复登录态，并在需要时重新请求用户信息。

`Vue Router` 中的 `meta.requiresAuth` 被用于保护投稿页、个人中心、收藏页和消息页。未登录用户访问这些路由时，会被重定向到登录页，并保留原始跳转地址。个人中心页面支持展示基础信息、上传头像、修改签名与昵称、修改密码，以及查看自己的投稿和观看历史。

#### 后端实现

后端认证功能主要由 `controller/auth.go`、`service/auth.go`、`service/user.go` 和相关 DAO 组成：

- 注册接口校验用户名、密码长度和邮箱格式，再由 `AuthService.Register` 创建用户。
- 密码不会明文存储，而是通过 `utils/bcrypt.go` 进行哈希处理。
- 登录接口验证账号密码后，使用 `utils/jwt.go` 生成 JWT Token，并返回 `accessToken + userInfo`。
- 用户信息接口通过 JWT 中间件解析出的 `user_id` 查询数据库，返回给前端。
- 头像上传走独立接口，文件内容会被读入后上传到 MinIO，再把头像地址更新到用户表。
- 个人资料编辑与密码修改走 `/users/me` 相关接口，由 `UserService` 执行业务逻辑。

#### 使用的技术

- `Gin`：提供注册、登录和用户资料接口
- `JWT`：生成和校验访问令牌
- `bcrypt`：密码加密存储
- `Pinia`：前端全局用户状态管理
- `Vue Router`：前端路由守卫
- `MinIO`：头像文件存储

#### 当前实现的局限

- 当前是单 Access Token 模式，没有 Refresh Token 机制，登录态续期能力有限。
- 配置文件中的 JWT Secret 以明文写在仓库里，不适合真实生产环境。
- 前端 `src/api/auth.ts` 中保留了 `getMe()` 方法，但实际用户信息恢复主要走 `/users/me`，接口设计上有一定重复和遗留痕迹。
- 注册被整合在登录页中，单独的 `RegisterView.vue` 当前并不是主路由实际使用页面。

### 4.2 视频上传、封面处理、异步转码与播放页

#### 功能作用

这是整个项目最核心的模块，负责完成从“用户选择视频文件”到“转码完成后可以在线播放”的完整链路。它决定了这个项目是否真正具备视频平台的实用价值。

#### 前端表现

投稿页 `UploadView.vue` 支持：

- 选择视频文件
- 校验视频格式与大小
- 录入标题、简介、分类
- 可选上传封面
- 展示上传中、转码中、完成或失败的状态
- 在编辑模式下修改已有视频信息

视频详情页 `VideoDetailView.vue` 会在进入页面后拉取详情数据。如果视频还未转码完成，页面会显示“转码进行中”的占位层，并定时轮询转码状态；一旦转码完成，播放器会切换为真实可播放状态。页面右侧还会加载一组关联视频用于推荐展示。

#### 后端实现

视频上传与转码链路主要集中在 `controller/video.go`、`service/video.go`、`consumer/transcode.go`、`utils/minio.go` 和 `utils/rabbitmq.go`。

当前实现步骤可以概括为：

1. 前端通过 `multipart/form-data` 上传视频文件、封面文件和文本字段。
2. 后端对视频扩展名和文件大小做基础校验。
3. 视频文件先被写入本地临时目录。
4. 如果系统安装了 `ffprobe`，会尝试提取视频时长。
5. 原始视频文件上传至 MinIO 的视频桶中。
6. 如果用户上传了封面，封面同样会上传到 MinIO；如果没有上传，则尝试调用 `ffmpeg` 从视频第 5 秒截帧生成封面。
7. 视频记录写入 MySQL，初始状态被标记为转码中。
8. 后端把转码任务序列化后投递到 RabbitMQ。
9. RabbitMQ 消费者收到任务后，会把原片从 MinIO 下载到本地临时目录。
10. 消费者通过 FFmpeg 执行 480p 转码，输出为 H.264 编码的 MP4 文件。
11. 转码后文件重新上传到 MinIO。
12. 数据库中的 `transcoded_url`、`transcode_status` 和发布状态会被更新。

播放页本身的详情接口还会做两件额外事情：

- 每次访问详情时递增视频播放量。
- 如果当前用户已登录，则记录一条播放历史。

#### 使用的技术

- `Gin`：文件上传与视频接口
- `GORM + MySQL`：视频记录、状态、统计字段存储
- `MinIO`：原片、转码后文件、封面对象存储
- `RabbitMQ`：转码任务异步队列
- `FFmpeg / ffprobe`：封面截帧、时长提取、视频转码
- `Axios + 轮询`：前端转码状态获取
- `DPlayer`：播放器组件

#### 当前实现的局限

- 当前只做单一清晰度转码，主要面向 480p，不支持多码率自适应播放。
- 上传接口主要依据扩展名判断文件类型，缺少更严格的文件头校验。
- 转码消费者执行 FFmpeg 时没有显式超时控制，异常视频可能拖住单次任务。
- 删除视频时会尝试删除对象存储中的文件，但没有更完整的失败补偿机制。
- 播放页“相关推荐”更多是从视频列表里取部分数据过滤，不是真正的推荐算法。

### 4.3 评论、二级回复与评论点赞

#### 功能作用

评论模块承担内容互动和讨论功能。项目没有只停留在“平铺评论”，而是支持一级评论与二级回复，这使它更接近真实视频社区的交流方式。

#### 前端表现

`CommentSection.vue` 提供了完整评论区组件，支持：

- 发表评论
- 回复某条评论或回复某条回复
- 评论列表切换“最新 / 最热”排序
- 评论点赞
- 删除自己的评论
- 展开更多回复

组件会在视频详情页中直接挂载，并把评论总数回传给父组件，用来更新页面上的评论统计。

#### 后端实现

评论核心逻辑位于 `controller/comment.go`、`service/comment.go` 和 `dao/comment.go`。

当前评论模型采用了典型的双字段结构：

- `parent_id`：直接回复的目标评论
- `root_id`：所属的一级评论

这种设计带来的好处是：

- 一级评论可以单独分页查询
- 二级回复可以按 `root_id` 聚合
- 回复关系既能表示“回复楼主”，也能表示“回复回复”

创建评论时，如果是回复场景，后端会先查父评论，再决定当前评论的 `root_id` 应该挂在哪个一级评论下面。查询评论列表时，后端先查一级评论，再为每条一级评论补充回复数和部分二级回复列表，同时为评论和回复拼装用户信息结构。评论点赞也有单独接口，点赞状态优先通过 Redis 判断，再配合数据库中的计数字段更新。

当回复发生时，如果被回复用户不是自己，后端还会异步创建一条通知记录。

#### 使用的技术

- `MySQL + GORM`：评论、回复关系和点赞计数
- `Redis`：评论点赞状态缓存
- `Vue` 组件化：评论区交互组织
- `goroutine`：回复通知异步创建

#### 当前实现的局限

- 评论列表查询过程中存在多次补查用户和父评论信息的行为，数据量变大后会有明显的查询放大风险。
- 评论点赞的 Redis 缓存目前更像轻量状态缓存，没有形成完整的批量回写机制。
- 删除评论后，前端主要通过本地数组更新 UI，后端是否需要联动维护视频评论数等统计，还可以进一步完善。

### 4.4 弹幕与 WebSocket 实时通信

#### 功能作用

弹幕系统是这个项目和普通视频列表项目拉开差距的重要功能。它实现的是“边播放边互动”的实时体验，而不是简单的评论替代。

#### 前端表现

视频详情页中叠加了 `DanmakuLayer.vue` 组件，负责：

- 根据视频 ID 创建 WebSocket 连接
- 拉取该视频已有弹幕数据
- 根据播放器当前时间点把对应秒数的弹幕渲染到屏幕上
- 在收到新的 WebSocket 消息时即时追加弹幕
- 处理暂停、继续播放、拖动进度时的弹幕展示逻辑
- 做一定次数的自动重连

用户在播放页下方输入弹幕内容后，会调用发送接口。发送成功时，当前页面会立刻本地显示新弹幕，而不是等待下一轮拉取。

#### 后端实现

后端弹幕功能主要由 `controller/danmaku.go`、`service/danmaku.go`、`dao/danmaku.go` 和 `utils/websocket.go` 构成。

实现思路是：

1. WebSocket 连接建立时，用户会根据 `video_id` 被加入对应的房间。
2. 后端内存中的 `DanmakuHub` 使用 `map[videoID]map[*websocket.Conn]bool` 保存房间与连接关系。
3. 用户发送弹幕时，后端先把弹幕写入数据库。
4. 写入成功后，通过 `broadcastDanmaku` 把新弹幕序列化为 JSON，再广播给同一个视频房间的所有连接。
5. 前端收到消息后即时插入屏幕。
6. 后端还提供按时间范围拉取弹幕的 HTTP 接口，前端首次进入视频页时会预加载一段弹幕池。

`DanmakuHub` 还带有心跳检测能力，可以定期清理失效连接，这说明项目作者考虑到了长连接的基本稳定性问题。

#### 使用的技术

- `WebSocket`
- `Gin`
- `MySQL`
- 内存房间管理与互斥锁
- 前端基于 DOM 动态渲染弹幕元素

#### 当前实现的局限

- 房间管理是单机内存实现，只适合单节点部署，无法天然支持多实例广播。
- 弹幕写入直接落 MySQL，在高并发场景下压力会比较集中。
- 当前没有看到更严格的弹幕内容审核、频率限制和风控体系。
- 前端一次性预加载固定时间范围弹幕，长视频场景下还可以继续优化。

### 4.5 视频点赞、收藏、通知与播放历史

#### 功能作用

这一组功能共同构成了平台中的“用户行为层”。它们决定用户能否对内容做反馈、留下痕迹、收到提醒，并形成连续使用体验。

#### 前端表现

视频详情页中提供了点赞、投币占位、收藏和分享按钮，其中点赞与收藏已经接入真实接口；通知页支持全部消息和未读消息切换、标记已读、全部已读和删除消息；个人中心中可以查看播放历史和我的视频，收藏页则可以查看当前收藏的视频内容。

#### 后端实现

**1. 视频点赞**

视频点赞核心在 `service/like.go`。其实现不是只写数据库，而是同时借助 Redis 做高频状态缓存：

- MySQL 中用 `video_likes` 记录用户是否点赞过某视频。
- Redis 中用 Hash 记录用户对视频的点赞状态。
- Redis 中用 String 记录某视频当前点赞数。
- Redis 中再用 Set 收集发生过点赞变化的视频 ID。
- 后台定时任务每 30 秒执行一次，把变化过的视频点赞数同步回数据库中的统计字段。

这套设计说明项目已经开始考虑“数据库记录”和“前台展示计数”分离的问题。

**2. 视频收藏**

收藏实现相对直接，由 `favorite` 模块完成：

- MySQL 中有独立收藏表保存用户与视频的关系。
- 前端可查询某视频是否已收藏。
- 收藏列表接口会把视频信息联表带出，用于收藏页展示。

**3. 消息通知**

通知模块支持三类触发场景：

- 评论被回复
- 评论被点赞
- 视频被点赞

通知由 `NotificationService` 在相关行为发生时异步创建，因此不会阻塞主流程。通知页能读取列表、统计未读数、标记单条已读、全部已读和删除通知。

**4. 播放历史**

播放历史在访问视频详情时自动记录。后端会先做去重，如果同一个用户此前已经看过该视频，则更新记录时间；如果是新记录则插入。同时，它还会清理超出上限或过旧的历史数据，说明这个模块不是简单追加，而是做了基本的数据维护。

#### 使用的技术

- `Redis`：点赞状态与计数缓存
- `MySQL`：收藏、通知、历史持久化
- `goroutine`：通知异步创建
- `Pinia + Axios`：前端状态同步

#### 当前实现的局限

- 收藏相对完整，但投币功能目前主要是前端占位，没有看到完整后端支持。
- 通知模块已能形成主链路，但消息类型仍然有限。
- 播放历史前端的“清空历史”当前主要是本地 UI 清空，没有看到对应的后端清空接口支撑。
- 个人中心中的部分统计值仍是硬编码展示，并不是由后端实时汇总得到。

### 4.6 首页列表、分类筛选、推荐与前端展示

#### 功能作用

首页承担的是“内容入口”的职责，包括推荐视频展示、分类筛选、列表展示和将用户导向详情页。虽然当前不具备复杂推荐算法，但已经形成了一个可浏览、可筛选的基本内容首页。

#### 前端表现

`HomeView.vue` 会在进入页面后拉取视频列表，并完成一次前端适配：

- 把后端返回的视频数据转换为卡片组件需要的结构
- 补充封面缺失时的 fallback 渐变色
- 展示相对时间、播放量、分区信息等
- 按当前分类切换列表

首页还配合 `NavBar`、`SideBar`、`BannerCarousel`、`VideoGrid`、`SectionBlock` 等组件，形成较完整的内容门户页面。整体视觉统一度较好，这也是当前前端比较突出的优势之一。

#### 后端实现

视频列表接口由 `VideoController.List` 提供，支持：

- 分页
- 分类参数
- 排序参数

后端会查询已发布视频，再额外查出视频作者信息，拼接后返回前端。这个接口已经足够支撑首页、相关推荐和部分个人页面的内容展示。

#### 使用的技术

- `Vue 3` 组件化页面组织
- `Vue Router`
- `Axios`
- 后端列表接口分页查询

#### 当前实现的局限

- 首页更多是内容展示层，尚未看到真正的推荐策略、热榜算法或搜索系统。
- 部分首页版块标题、榜单和分类映射带有明显的演示性质。
- 当前未看到首页列表的 Redis 缓存，随着访问量增加，数据库压力会逐步上升。

## 6. 项目目录与分层职责

### 6.1 目录结构理解

项目整体可以分成前端、后端和部署辅助材料三部分：

- `bilibili-frontend`：Vue 前端项目
- `bilibili-backend`：Go 后端项目
- `scripts`、`nginx` 以及根目录说明文件：部署或资料补充

### 6.2 后端分层职责

后端目录体现了较明显的分层设计：

| 目录 | 职责 |
| --- | --- |
| `controller` | 接收 HTTP 请求、参数校验、组织响应 |
| `service` | 核心业务逻辑编排 |
| `dao` | 数据访问逻辑 |
| `model` | 数据表与实体定义 |
| `middleware` | 鉴权、跨域等中间件 |
| `consumer` | 异步消费逻辑 |
| `utils` | JWT、Redis、MinIO、RabbitMQ、WebSocket 等基础工具 |
| `router` | 路由注册 |

`main.go` 中的初始化流程也比较直观：先加载配置，再连接数据库，随后初始化 MinIO、RabbitMQ、Redis，之后手动组装 DAO、Service、Controller，最后注册路由。这种显式依赖注入方式虽然不算高级，但阅读成本低，非常适合项目讲解。

### 6.3 前端分层职责

这说明项目并不是把全部逻辑堆在页面里，而是已经具备基本工程拆分意识。

## 7. 当前项目的亮点

### 7.1 主业务闭环比较完整

很多仿站项目通常只做到首页、详情页和登录页，而这个项目真正打通了“上传 - 存储 - 转码 - 播放 - 互动 - 通知”的核心链路。对于一个学习型项目来说，这已经不是简单的页面还原，而是带有后端系统设计思路的实现。

### 7.2 技术选型贴合视频平台场景

引入 MinIO、RabbitMQ、FFmpeg 和 WebSocket 都不是为了堆技术栈，而是都能在业务里找到直接落点：

- MinIO 解决文件对象存储
- RabbitMQ 解决上传与转码解耦
- FFmpeg 解决转码和封面截图
- WebSocket 解决弹幕实时广播

这种“技术和业务一一对应”的特征，使项目更适合用于讲述技术决策。

### 7.3 前端视觉统一度较高

前端不是默认样式拼接，而是有比较稳定的科技风视觉语言，包括：

- 深色背景
- 青紫色发光元素
- 等宽字体与终端式文案
- 统一的卡片、按钮和输入框风格

这会让项目在演示时比普通 CRUD 项目更有辨识度。

### 7.4 分层结构清晰，便于继续扩展

虽然项目还没有引入更复杂的工程工具，但当前后端的分层已经足够支持后续继续加功能，例如搜索、关注、审核、后台管理、多清晰度转码等，不需要整体推倒重来。

## 8. 当前观察到的不足与改进建议

这一部分非常重要，因为它决定这份介绍文档是不是足够诚实、专业。

### 8.1 配置安全性需要立即改进

当前 `bilibili-backend/config/config.yaml` 中存在数据库、Redis、RabbitMQ、MinIO 和 JWT 的明文配置。对展示项目来说这可能是为了方便运行，但从工程安全角度看，这是一个明确缺点。

建议：

- 敏感信息改为环境变量注入
- 仓库中保留 `config.example.yaml`
- 真实配置文件加入忽略规则

### 8.2 前端部分展示仍有占位和硬编码

从当前代码可以看到，项目虽然主链路打通，但部分页面仍有演示性质：

- 个人中心部分统计值是固定数字，不是接口实时汇总
- 首页一些榜单和内容块带有明显前端生成性质
- 投币按钮当前更多是交互占位，不是完整业务
- 仓库中存在若干未完全接入路由的页面文件

建议：

- 把个人中心统计改成真实接口聚合
- 为首页增加真实热度排序、搜索与推荐能力
- 清理未接入页面，或继续补完其路由和 API

### 8.3 缓存与数据一致性还有提升空间

点赞模块已经开始使用 Redis，但还可以继续完善：

- 增加 Redis 重建策略
- 更严格地处理缓存与数据库不一致问题
- 增强异常情况下的补偿逻辑

评论点赞和首页列表也可以进一步引入更完整的缓存机制。

### 8.4 异步转码链路还可以更稳

当前转码链路已经可用，但距离生产级还有差距：

- 缺少更明确的任务失败重试和死信处理
- 缺少转码超时控制
- 只有单清晰度输出
- 缺少任务监控和更完整日志

建议：

- 为消费者增加任务超时与失败重试
- 引入多码率转码
- 统一记录转码日志与错误原因

### 8.5 实时系统仍是单机思路

弹幕房间依赖后端内存结构保存连接关系，这意味着它适合单实例部署，不适合直接扩展到多节点。

建议：

- 通过 Redis Pub/Sub 或消息总线实现跨节点广播
- 为弹幕发送增加频率限制与内容审核

### 8.6 测试与可维护性仍偏薄弱

从当前仓库结构来看，项目的重点放在功能实现上，但自动化测试、日志规范、统一错误追踪等方面仍然较弱。

建议：

- 为核心 Service 增加单元测试
- 为关键接口增加集成测试
- 把日志从简单输出提升到结构化日志
- 引入更稳定的配置和环境分层机制

## 9. 适合展示或答辩时的总结性描述

如果要把这个项目用于课程汇报、答辩或面试介绍，可以把它概括为下面这段话：

这个项目是一个基于 Go 和 Vue 3 实现的仿 B 站视频平台，重点不是单纯还原页面，而是完整实现了视频网站的核心业务闭环。系统采用前后端分离架构，前端负责首页、详情页、投稿页、个人中心和互动页面展示，后端负责用户认证、视频管理、评论弹幕、点赞收藏、消息通知和播放历史等业务。项目中引入了 MinIO、RabbitMQ、FFmpeg、Redis 和 WebSocket，用于解决对象存储、异步转码、实时弹幕和高频交互状态缓存等问题。当前版本已经具备较完整的主流程，同时在配置安全、前端数据真实化、多清晰度转码和多节点实时通信等方面仍有进一步优化空间。

如果用一句更简洁的话总结，这个项目的价值在于：它已经从“仿站页面”进阶到了“有工程化思路的视频平台原型”。
