一、数据库设计

在 `bilibili` 库中创建/调整 `videos` 表：

- `id`：BIGINT 自增主键
- `user_id`：BIGINT，关联 `users.id`，非空
- `title`：VARCHAR(200)，非空
- `description`：TEXT，视频简介
- `cover_url`：VARCHAR(255)，封面图 URL（MinIO）
- `video_url`：VARCHAR(255)，原片 URL（MinIO，存档用，不直接播放）
- `transcoded_url`：VARCHAR(255)，转码后视频 URL（MinIO，480P 播放地址），转码完成前为空
- `category`：VARCHAR(50)，分类标识（模型自行设计分类体系）
- `status`：TINYINT，默认 2（0=待审核预留，1=已发布，2=转码中）。管理员审核功能暂未实现，转码完成后自动将 `status` 设为 1（已发布），无需人工审核
- `transcode_status`：TINYINT，默认 0（0=未开始，1=转码中，2=转码完成，3=转码失败）
- `view_count`：BIGINT，默认 0
- `like_count`：INT，默认 0
- `comment_count`：INT，默认 0
- `danmaku_count`：INT，默认 0
- `created_at` / `updated_at`：TIMESTAMP

索引：`(category, status, created_at)`、`(user_id, created_at)`。

> 因为只转码单档清晰度，无需独立的 `video_qualities` 多清晰度关联表，转码后的 480P（或 360P）地址直接存入 `videos.transcoded_url` 字段，简化设计。

二、后端 API 设计

统一响应格式 `{ code, message, data }`，code=200 为成功。

1. 视频上传
- POST `/api/v1/videos`
- Content-Type：`multipart/form-data`
- 字段：
  - `file`：视频文件，格式限制由模型自行合理设定（如 `mp4/mov/webm/mkv` 等），文件大小限制由模型自行判断并说明
  - `title`：视频标题，必填，长度限制由模型合理设计
  - `description`：视频简介，可选
  - `category`：分类标识，必填
  - `cover`：封面图片文件，可选。如未上传，后端调用 FFmpeg 截取视频第 5 秒画面作为默认封面，上传至 MinIO `covers/` 路径

- 业务逻辑：
  1. JWT 鉴权获取 `user_id`
  2. 校验文件格式、大小（模型自行设定合理限制）
  3. 生成唯一文件名（UUID），原片直接上传到 MinIO（路径如 `videos/raw/UUID.mp4`，模型自行设计合理的 MinIO 路径结构）
  4. 封面处理：用户上传则转存 MinIO `covers/`；未上传则 FFmpeg 截取第 5 秒帧，转为 jpg/png 上传至 `covers/`
  5. 写入 `videos` 表：`status=2`（转码中），`transcode_status=1`（转码中），记录 `video_url`（原片地址）
  6. 向 RabbitMQ 发送转码任务消息：队列名模型自行定义（如 `video_transcode`），消息体至少包含 `video_id` 和 `input_url`（MinIO 原片可访问地址或本地临时路径，由模型自行设计）
  7. 返回视频基本信息（id, title, cover_url, status, transcode_status, created_at）

- 错误码：40001（参数错误）、40003（格式不支持）、40004（文件过大）

2. 查询转码状态（前端轮询用）
- GET `/api/v1/videos/:id/transcode`
- 返回：`{ transcode_status, transcoded_url? }`
- `transcode_status`：0=未开始，1=转码中，2=转码完成（此时 `transcoded_url` 有值），3=转码失败

3. 视频列表
- GET `/api/v1/videos?page=1&size=20&category=xxx&sort=new`
- 参数：分页、分类筛选（可选）、排序（`new` 最新 / `hot` 最热）
- 只返回 `status=1`（已发布）且 `transcode_status=2`（转码完成）的视频
- 每条数据包含：`id, title, cover_url, category, view_count, like_count, created_at`，以及 UP 主信息 `user: {id, username, avatar}`

4. 视频详情
- GET `/api/v1/videos/:id`
- 返回视频完整信息：含 `title, description, cover_url, video_url（原片）, transcoded_url（480P 播放地址）, user_info, view_count, like_count, created_at`
- 业务逻辑：`view_count + 1`（直接 MySQL +1 或 Redis，个人项目建议直接 +1 简化）

5. 删除视频
- DELETE `/api/v1/videos/:id`
- 校验：只能删除自己上传的视频
- 业务逻辑：删除数据库记录 + 删除 MinIO 上对应文件（原片 + 转码后视频 + 封面）

三、FFmpeg + RabbitMQ 异步转码流程（核心）

RabbitMQ 连接
- 使用 Go 的 RabbitMQ 客户端库（如 `github.com/rabbitmq/amqp091-go`）
- 连接地址从环境变量读取（`RABBITMQ_HOST` 等，与 Docker Compose 中的 RabbitMQ 服务对应）

生产者（上传 API 中）
- 视频入库后，将转码任务序列化为 JSON 发送到指定队列
- 消息格式模型自行设计，必须包含：`video_id`（数据库 ID）、`input_path` 或 `input_url`（原片位置）

消费者（后端独立启动或随服务启动）
- 服务启动时初始化 RabbitMQ 消费者，监听转码队列
- 消费到消息后：
  1. 从 MinIO 下载原片到本地临时目录（或如果原片在本地则直接读取）
  2. 调用 FFmpeg 转码为 480P（默认），如果服务器压力大则降级为 360P。只输出单档清晰度，不需要多档
  3. FFmpeg 参考参数（480P）：`ffmpeg -i input.mp4 -vf scale=-2:480 -c:v libx264 -preset fast -crf 23 -c:a copy -movflags +faststart output_480p.mp4`
  4. 转码完成后，将输出文件上传回 MinIO（路径如 `videos/480p/UUID_480p.mp4`，模型自行设计）
  5. 更新数据库：`videos.transcoded_url = MinIO地址`、`transcode_status = 2`、`status = 1`
  6. 如果转码失败，更新 `transcode_status = 3`，可记录失败原因到日志
  7. 清理本地临时文件

转码策略
- 单档输出：仅 480P（或 360P），不生成 720P/1080P 等多档
- 转码后的文件作为唯一播放源，原片 `video_url` 仅作存档/备用
- 消费者需具备基本的错误处理和重试机制（至少记录日志，个人项目可先不做死信队列）

四、前端页面设计

页面一：投稿页（UploadView）

延续深空科技风，标准三栏布局。

上传核心区域
- 顶部标题：`// UPLOAD_PROTOCOL // v1.0`，等宽字体霓虹青，带闪烁光标 `_`
- 文件选择区（主要方式）：
  - 高度 280px，边框 2px 虚线 `rgba(0,240,255,0.2)`，圆角 4px，背景极夜黑
  - 中央显示一个明显的 `[ SELECT_FILE ]` 按钮（电光紫边框，hover 霓虹青，扫描线扫过），等宽字体
  - 按钮下方文字：`> CLICK_TO_UPLOAD` 或 `SUPPORTED: MP4 / MOV / WEBM / MKV`，等宽字体暗灰
  - 点击按钮唤起隐藏的 `<input type="file" accept="视频格式" />`
- 拖拽上传（可选实现）：
  - 如果模型判断拖拽容易实现，则支持拖拽文件到选择区
  - 拖拽进入时：边框变实线霓虹青，外散发光阴影，背景扫描线纹理，中央文字变 `> READY_TO_UPLOAD`
  - 如不易实现，可仅保留点击上传，不做强制要求
- 选择文件后：显示文件信息卡片
  - 左侧：60×40 视频缩略图占位（纯色块，按格式显示不同颜色：mp4=霓虹青、mov=电光紫、webm=极光蓝）
  - 右侧：文件名（等宽字体，冷白）、文件大小（格式化，如 `> 124.5 MB`）、视频时长（如有，等宽字体）
  - 右侧删除按钮：`[X]` 极小，红色，hover 故障闪烁

- 进度条：4px 高，背景 `#14161f`，填充电光紫→霓虹青渐变，带 shimmer 斜向光条流动

- 取消按钮：`[ ABORT ]`，红色边框，hover 故障闪烁

转码状态（上传完成后进入此阶段）
- 上传完成后，进度条变为青色脉冲呼吸动画（opacity 0.4 ↔ 1.0 循环）
- 中央显示：`> TRANSCODING // 480P...`，等宽字体，霓虹青，末尾带闪烁光标
- 下方可显示极简状态日志（终端风格，逐行打印模拟）：`[INFO] DECODING...` `[INFO] RESCALING...` `[INFO] ENCODING_H264...`，暗灰色等宽字体，当前执行行高亮为青色
- 转码完成后：进度条闪烁一次纯青色，显示 `> TRANSCODE_COMPLETE // READY_FOR_STREAM`，1.5 秒后自动跳转视频详情页 `/video/:id`
- 转码失败：进度条变红，显示 `> TRANSCODE_FAILED // CHECK_LOGS`，下方 `[ RETRY ]` 按钮（电光紫）

视频信息表单（选择文件后显示）
垂直排列，间距 28px：
- 标题：标签 `> TITLE`，输入框暗背景紫边框聚焦青，占位符 `> 输入视频标题...`，字数统计 `0/200`
- 分类选择：标签 `> CATEGORY`，分类按钮横向排列（模型自行设计分类），未选中暗灰，选中霓虹青边框+青色背景+`■` 指示器，瞬时切换 0.1s

- 封面选择：标签 `> COVER_ART`，默认 `> AUTO_GENERATE`（后端 FFmpeg 截取第 5 秒），可选上传图片，选中后显示 60px×40px 直角预览图，可 `> RESET`
- 提交按钮：`[ EXECUTE_UPLOAD ]`，电光紫→hover 霓虹青，扫描线扫过。点击后禁用，文字变为 `> UPLOADING...` → 完成后变为 `> TRANSCODING...`

五、前端状态与交互（Pinia 扩展）

- `uploadVideo(formData)`：调 POST `/api/v1/videos`，FormData 含 file/title/description/category/cover。返回视频 ID。
- `pollTranscodeStatus(videoId)`：轮询 GET `/api/v1/videos/:id/transcode`，每 3 秒一次，直到 `transcode_status=2`（完成）或 `3`（失败）。用于投稿页和详情页。
- `fetchVideoDetail(id)`：调视频详情 API。
- `deleteVideo(id)`：调删除 API。

路由
- 投稿页：`/upload`，命名 `upload`，需登录（路由守卫拦截）
- 视频详情页：`/video/:id`，命名 `videoDetail`
